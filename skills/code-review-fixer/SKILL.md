# 代码审查问题修复 (Code Review Fixer)

根据刚完成的代码审查报告，与开发者确认并逐条修复问题。

---

## 前提条件

本 skill 需要在 `layered-code-review` 执行完毕后调用，依赖其输出的审查报告。
若在对话历史中未找到审查报告，先提示用户执行 `/layered-code-review`。

---

## 执行步骤

### Step 1 — 提取可修复问题

从审查报告中提取 **🔴 必须修改** 和 **🟡 建议考虑** 两个区块的所有问题，生成编号清单：

```
以下问题可自动处理，请确认你希望修复哪些（回复编号，逗号分隔；或"全部"/"跳过"）：

  🔴 1. [文件名 第N行] 问题简述
  🔴 2. [文件名 第N行] 问题简述
  🟡 3. [文件名 第N行] 问题简述
  ...

⚠️  以下问题需要你手动处理（涉及业务判断，AI 不适合自动修改）：
  - [说明原因]
```

**等待用户回复。**

以下类型的问题**不纳入自动修复清单**，标注"需手动处理"：
- 设计层面的重构建议（涉及跨文件重构）
- 业务逻辑不明确的问题（修改可能引入新 bug）
- 需要删除代码 / 删除文件的操作

### Step 2 — 逐条修复

对用户确认的每条问题：
1. 读取涉及的完整文件，理解上下文
2. 按审查报告中的"修改建议"进行最小化修改（只改报告指出的问题，不做额外重构）
3. 修改前展示 diff 预览，确认无误后写入文件
4. 修改完成后输出进度：`✅ [N/总数] 已修复：问题简述`

### Step 3 — 完成汇总

所有修改完成后输出：

```
======================================
✅ 修复完成
======================================
已修复 N 条问题：
  ✅ 1. 问题简述
  ✅ 2. 问题简述

未修复（已跳过）：
  ⏭️  3. 问题简述（原因）

建议执行 /peer-review 进行复核。
======================================
```

---

## 约束

- 每次修改只针对报告中指出的具体问题，不进行范围外的改动
- 修改前必须读取文件内容，不凭记忆修改
- 有疑义的修改优先询问，而不是猜测
