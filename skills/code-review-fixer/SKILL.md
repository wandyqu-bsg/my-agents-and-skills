# 代码审查问题修复 (Code Review Fixer)

根据刚完成的代码审查报告，与开发者确认并逐条修复问题。

---

## 前提条件

本 skill 需要在 `layered-code-review` 执行完毕后调用，依赖其输出的审查报告。

- 若对话历史中存在 `layered-code-review` 的输出报告，直接使用
- 若未找到，提示用户先执行 `/layered-code-review`，并停止后续步骤

---

## 执行步骤

### Step 1 — 提取可修复问题

从审查报告中扫描以下区块，按优先级顺序提取问题：

| 区块 | 处理方式 |
|---|---|
| 🔴 **必须修改** | 纳入自动修复清单（高优先级） |
| 🟡 **建议考虑** | 纳入自动修复清单（中优先级） |
| 💭 **改了更好** | 纳入可选修复清单（低优先级，可整体跳过） |
| 💅 **代码规范（Clean Code / PMD）** | 纳入可选修复清单（低优先级，纯规范类） |
| 📋 **测试覆盖** | **不纳入**，始终标注"需手动处理"：引导使用 `/ut-generator` |

> **优先利用 `⬇️ 下一步行动` 编号列表**：若报告末尾存在此区块，直接复用其编号和简述，避免重复整理。

**分类标注规则（以下问题纳入"需手动处理"，不进入自动修复）：**

- **Layer 0 安全类**：硬编码密钥、注入风险等安全修改须人工确认，AI 可描述改法但不自动写入
- **跨文件重构**：涉及接口修改、类层次调整等影响面广的设计建议
- **业务逻辑不明确**：报告中有疑问或"根据业务判断"字样的建议
- **删除操作**：需要删除代码、方法或文件的操作
- **📋 测试覆盖**：引导使用 `/ut-generator` 生成，不在此 skill 中处理

向用户输出清单：

```
以下问题可自动处理，请确认你希望修复哪些
（回复编号，逗号分隔；"全部"；"跳过可选"仅修 🔴🟡；或"跳过"）：

  🔴 1. [文件名 第N行] 问题简述
  🔴 2. [文件名 第N行] 问题简述
  🟡 3. [文件名 第N行] 问题简述
  💭 4. [文件名 第N行] 问题简述（可选）
  💅 5. [文件名 第N行] 问题简述（可选）

⚠️  以下问题需要你手动处理：
  - [📋 测试覆盖] 建议使用 /ut-generator 生成测试骨架
  - [🔐 安全类] 文件名 第N行：[问题描述]，请人工确认修改方案
  - [设计重构] 文件名：[问题描述]（涉及跨文件改动，AI 不适合自动处理）
```

**等待用户回复。**

---

### Step 2 — 逐条修复（含安全检查）

对用户确认的每条问题，**在修改前必须完成安全检查**：

#### 2.1 修复前安全检查

读取涉及文件的完整内容（不只读问题行），检查以下项目：

1. **调用方影响**：修改的方法签名或行为是否被其他类调用？若是，确认调用方不受影响
2. **相邻逻辑风险**：修改点周围是否有 null 检查、try/catch、循环边界等上下文，修改后是否引入新的 NPE / 异常风险？
3. **最小化原则**：只改报告明确指出的代码行，不顺手重构周边代码
4. **副作用检查**：修改是否改变了现有方法的外部行为（返回值、抛出异常类型、状态变化）？

> 若安全检查发现风险，在该条问题前输出 ⚠️ 警告并说明风险，询问用户是否继续。

#### 2.2 执行修改

1. 按审查报告中的"修改建议"进行最小化修改
2. **复杂修改（涉及逻辑变更、多处改动）必须展示 diff 预览**，确认后再写入；简单修改（拼写、命名、格式）可直接写入
3. 写入完成后输出进度：`✅ [N/总数] 已修复：问题简述`

---

### Step 3 — 完成汇总

所有修改完成后输出：

```
======================================
✅ 修复完成
======================================
已修复 N 条问题：
  ✅ 1. 问题简述
  ✅ 2. 问题简述

未修复（已跳过）：
  ⏭️  3. 问题简述（用户跳过）

需手动处理：
  🔧 4. [📋 测试覆盖] 建议执行 /ut-generator
  🔧 5. [设计重构] 问题简述（需人工判断）

建议执行 /peer-review 进行复核，确保修复未引入新问题。
======================================
```

---

## 约束

- 每次修改只针对报告中指出的具体问题，不进行范围外的改动
- 修改前必须读取文件完整内容，**不凭记忆修改**
- 安全类问题（🔐 Layer 0）不自动写入，需人工确认
- 有疑义的修改优先询问，而不是猜测
- 禁止为了"顺手优化"而修改报告范围外的代码
