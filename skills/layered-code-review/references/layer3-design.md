# Layer 3 — 设计与架构合理性

> Layer 3 要求**主动读取相关类**，不能只看 diff 本身。这一层的发现通常是最有价值的，也是普通 review 工具做不到的地方。

---

## 上下文读取策略（按优先级）

执行以下步骤收集上下文，**遇到超大类时优先读方法签名和 Javadoc，不必全文读完**：

### 1. 读取改动类的父类 / 实现的接口
- 改动的方法是否是 `@Override`？如果是，父类/接口的约定是什么？
- 覆写是否保持了 Liskov 替换原则（子类行为是否兼容父类约定）？
- 接口的 `default` 方法和实现类的覆写：哪些类可能没有覆写，默认行为是否安全？

### 2. 读取调用了改动方法的类
- 谁在调用被改动的方法？（用 grep / semantic search 找 usages）
- 调用方是否依赖了方法的旧行为（返回值、是否抛异常、副作用）？
- 新增的行为是否对所有调用方都是安全的，还是只对部分调用方有意义？

### 3. 读取同 package 下的同级类（风格对比）
- 同类问题在其他类里是怎么解决的？
- 项目中是否已有类似的 helper/util，新改动是否造成了重复？

## 审查要点

### 继承与多态
- **Override 一致性**：新增/修改的 `@Override` 方法，其行为是否与父类契约一致？
  - 父类约定：空集合 → 返回空 list；子类实现：返回 null？→ 🔴 不一致
- **default 方法陷阱**：接口中新增 `default` 方法，所有已知实现类是否都能接受默认行为？对于非 Product 类型，默认 `return false` 是否正确？
- **不必要的暴露**：将 `protected final` 方法改为 `protected static`，会改变其语义。`static` 意味着方法不依赖实例状态，是否真的如此？

### 职责合理性
- **单一职责**：改动是否让某个类承担了太多职责？（如在 entity 中做持久化操作 `getPersistenceManager().updateEntity()`，而非在 service 层）
- **改动范围合理性**：改动是否影响了不相关的路径？（如在不相关的 `else` 分支里也加了同样的逻辑）
- **重复逻辑**：同一段代码在多处重复（如同一个 `updateFundCurrency` 调用出现在同一方法的两个分支），是否可以合并？

### 与项目风格的一致性
- **实体类中的业务逻辑**：项目中其他实体类是否也在 entity 中直接调用 persistence 和 poisoner？还是统一在 service/manager 层处理？
  - 如果其他 entity 没有这种模式，新加的代码就是风格异类，需要说明理由或建议移到 service 层
- **前端交互模式**：新增的 `confirm()` 弹窗是否与项目中其他敏感操作的交互方式一致？（其他类似功能有没有用弹窗、Toast、或独立确认页面？）
- **常量提取**：项目中有没有统一的常量类？新加的字符串常量（如 `"PortfolioFundBean.currency"`）是否应该放到那里？

### API 设计
- **接口变更**：对公开接口（interface）中的 `default` 方法新增，是否会破坏已有的实现类（泛型约束、接口继承体系）？
- **static 方法的语义**：某个方法被改为 `static` 是否有充分理由？`static` 方法不能被子类覆写，这会影响可测试性（无法 mock）
- **参数和返回值**：新方法的签名是否与项目中同类方法的风格一致（参数顺序、命名、是否可为 null）？

### 前端组件（JS / React）
- 新增的函数（如 `hasProductCurrencyField`, `checkProductCurrencyChange`）是否与已有的同类函数（如 `hasOrganizationNameField`, `checkNameChange`）保持一致的命名和调用模式？
- 全局函数定义（在 `<script>` 中定义的函数）是否存在命名冲突风险？
- 组件状态管理：是否与项目中其他组件的 state 管理方式一致（Redux / Context / local state）？

## 常见设计问题速查

| 问题模式 | 严重性 | 建议 |
|---|---|---|
| Entity 中直接调用 DAO / persistence | 🟡 | 考虑移到 service/manager 层 |
| 接口 default 方法默认行为不安全 | 🟡 | 加 Javadoc 说明默认值的含义 |
| 同一方法被调用两次（可能重复） | 🟡 | 确认是否可合并，或加注释说明意图 |
| static 方法替代了本应可覆写的实现 | 💭 | 评估对可测试性的影响 |
| 字符串字面量未提取为常量 | 💭 | 提取为命名常量，集中管理 |
| 新增逻辑与同类代码风格不一致 | 💭 | 对齐已有模式或提 issue 讨论 |
